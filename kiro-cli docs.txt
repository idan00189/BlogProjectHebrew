## CLI COMMAND: kiro-cli chat

### Description
Starts an interactive chat session with the Kiro AI agent. If an input is provided, it processes the question directly.

### Syntax
`kiro-cli chat [OPTIONS] [INPUT]`

### Parameters
#### Options
- **--no-interactive** (flag) - Optional - Print first response to STDOUT without interactive mode
- **--resume** / **-r** (flag) - Optional - Resume the previous conversation from this directory
- **--resume-picker** (flag) - Optional - Open interactive session picker to choose which session to resume
- **--list-sessions** (flag) - Optional - List all saved chat sessions for the current directory
- **--list-models** (flag) - Optional - Display available models
- **--delete-session** (ID) - Optional - Delete a saved chat session by ID
- **--agent** (string) - Optional - Specify which agent to use

### ACP Initialize Connection

Source: https://kiro.dev/docs/cli/acp

The initialize method is the first step in the ACP lifecycle, used to exchange protocol versions and capabilities between the client editor and the Kiro agent.

```APIDOC
## POST initialize

### Description
Initializes the connection between the editor and the Kiro agent, exchanging capabilities and protocol information.

### Method
POST (JSON-RPC)

### Endpoint
initialize

### Parameters
#### Request Body
- **protocolVersion** (number) - Required - The ACP protocol version supported by the client.
- **clientCapabilities** (object) - Required - Capabilities supported by the editor (e.g., fs, terminal).
- **clientInfo** (object) - Required - Metadata about the client editor (name, version).

### Request Example
{
  "jsonrpc": "2.0",
  "id": 0,
  "method": "initialize",
  "params": {
    "protocolVersion": 1,
    "clientCapabilities": {
      "fs": { "readTextFile": true, "writeTextFile": true },
      "terminal": true
    },
    "clientInfo": {
      "name": "my-editor",
      "version": "1.0.0"
    }
  }
}

### Response
#### Success Response (200)
- **protocolVersion** (number) - The agreed protocol version.
- **agentCapabilities** (object) - Capabilities supported by the Kiro agent.
- **agentInfo** (object) - Metadata about the Kiro agent.

#### Response Example
{
  "jsonrpc": "2.0",
  "id": 0,
  "result": {
    "protocolVersion": 1,
    "agentCapabilities": { "loadSession": true, "promptCapabilities": { "image": true } },
    "agentInfo": { "name": "kiro-cli", "version": "1.5.0" }
  }
}
```

### Bash Script to Handle Kiro CLI Exit Codes

Source: https://kiro.dev/docs/cli/reference/exit-codes

A bash script that executes a Kiro CLI command and then uses a case statement to handle different exit codes. It specifically checks for success (0), MCP startup failure (3), and other general failures, providing appropriate feedback or exiting with the relevant code.

```bash
#!/bin/bash
kiro-cli chat --require-mcp-startup --no-interactive --trust-all-tools "Run analysis"
exit_code=$?

case $exit_code in
    0) echo "Success" ;;
    3) echo "MCP servers failed to start"; exit 1 ;;
    *) echo "Failed with code $exit_code"; exit $exit_code ;;
esac
```

```APIDOC
## Tools Field

The `tools` field lists all tools that the agent can potentially use. Tools include built-in tools and tools from MCP servers.

### Tool Specification

- **Built-in tools**: Specified by their name (e.g., `read`, `shell`).
- **MCP server tools**: Prefixed with `@` followed by the server name (e.g., `@git`).
- **Specific MCP server tool**: Use `@server_name/tool_name` (e.g., `@rust-analyzer/check_code`).
- **Wildcard for all tools**: Use `*` to include all available tools (built-in and from MCP servers).
- **Wildcard for built-in tools**: Use `@builtin` to include all built-in tools.
- **Wildcard for MCP server tools**: Use `@server_name` to include all tools from a specific MCP server.

### Example: Specifying Tools
```json
{
  "tools": [
    "read",
    "write",
    "shell",
    "@git",
    "@rust-analyzer/check_code"
  ]
}
```

### Example: Including All Tools
```json
{
  "tools": ["*"]
}
```
```

### ToolAliases Field

Source: https://kiro.dev/docs/cli/custom-agents/configuration-reference

Allows remapping tool names to resolve naming collisions or create more intuitive names.

```APIDOC
## ToolAliases Field

The `toolAliases` field is an advanced feature that allows you to remap tool names. This is primarily used to resolve naming collisions between tools from different MCP servers, or to create more intuitive names for specific tools.

### Example: Resolving Naming Collisions

If both `@github-mcp` and `@gitlab-mcp` servers provide a tool called `get_issues`, you would have a naming collision. You can use `toolAliases` to disambiguate them:
```json
{
  "toolAliases": {
    "@github-mcp/get_issues": "github_issues",
    "@gitlab-mcp/get_issues": "gitlab_issues"
  }
}
```

With this configuration, the tools will be available to the agent as `github_issues` and `gitlab_issues` instead of having a collision on `get_issues`.

### Example: Creating Shorter or More Intuitive Names

You can also use aliases to create shorter or more intuitive names for frequently used tools:
```json
{
  "toolAliases": {
    "@aws-cloud-formation/deploy_stack_with_parameters": "deploy_cf",
    "@kubernetes-tools/get_pod_logs_with_namespace": "pod_logs"
  }
}
```

The key is the original tool name (including server prefix for MCP tools), and the value is the new name to use.
```

### AllowedTools Field

Source: https://kiro.dev/docs/cli/custom-agents/configuration-reference

Specifies tools that can be used without prompting the user for permission, enhancing security.

```APIDOC
## AllowedTools Field

The `allowedTools` field specifies which tools can be used without prompting the user for permission. This is a security feature that helps prevent unauthorized tool usage.

### Example: Specifying Allowed Tools
```json
{
  "allowedTools": [
    "read",
    "write",
    "@git/git_status",
    "@server/read_*",
    "@fetch"
  ]
}
```

### Allowed Tool Patterns

- **Exact matches**:
  - Built-in tools: `"read"`, `"shell"`, `"knowledge"`
  - Specific MCP tools: `"@server_name/tool_name"` (e.g., `"@git/git_status"`)
  - All tools from MCP server: `"@server_name"` (e.g., `"@fetch"`)

- **Wildcard patterns**:
  The `allowedTools` field supports glob-style wildcard patterns using `*` and `?`.
  - Example: `"@server/read_*"` allows all tools starting with `read_` from the `@server` MCP server.
```
